What this is

webcurator-gym is a Prime Lab / Verifiers workspace whose centerpiece is the pretrain-data-curator environment (environments/pretrain_data_curator/). An LLM agent is asked to curate an LLM pretraining dataset from the pre-cutoff Hugging Face universe under a fixed token budget. The curated mixture is then scored by training a fixed GPT-2-scale proxy student on it — everything except the data is held constant — and measuring cross-entropy on a held-out validation stream, minus a contamination penalty: 

reward R(M, H) = alpha_perf · Perf(M)  −  lambda_leakage · Leakage(M, H)

Perf     proxy-student cross-entropy vs. a neutral baseline, power-law scaled so the
         NanoGPT-speedrun target loss (3.28 nats/token) maps to exactly 1.0
Leakage  token-weighted n-gram contamination (decon) against public benchmarks
         and the held-out validation set

The agent gets no bespoke tools — just one prompt and its harness's own shell (bash, Codex, mini-SWE-agent, …) with the hf CLI available. It searches and inspects datasets, optionally self-scores drafts, and finishes by writing a manifest JSON describing the mixture. Every run on the leaderboard is one such rollout with a 400M-token budget and real GPU proxy-student scoring.

Anatomy of a rollout

What happens between "run eval" and a reward landing on the leaderboard.

  1. Setup. CuratorTaskset.setup() validates HF_TOKEN, renders the single curation task (tasks.py) as the initial user prompt, and copies a leakage-safe self_score.py helper into the agent's workspace.
  2. Agent loop. The agent explores the Hub from its shell (hf datasets ls --search …, hf datasets info …), applies its own judgment about quality and mixing, and may run python self_score.py draft.json to estimate proxy CE and token fill for a draft manifest (screened against public benchmarks only — never the hidden val set).
  3. Finalize. Writing /workspace/manifest.json is the completion signal: a token budget plus weighted sources (id, weight, optional config/split/ text_field/filters/caps, or kind: "local" files prepared in the workspace). finalize() parses it, with trace-based recovery if the file is malformed.
  4. Materialize. CorpusBuilder streams each source from the Hub (or capped local files) straight to disk, applying structured filters, per-source caps, weights, and the global token budget. Fetches are cached per rollout so previews and scoring see identical documents.
  5. Score. CuratorScorer computes the two reward terms: a proxy-student trainer produces Perf, and the vendored decon binary screens the corpus for benchmark/val-set contamination to produce Leakage. Both, plus diagnostics (budget_fill_ratio, corpus_tokens, tool_error_count, external_failure, …), are emitted as run metrics.
  6. Proxy-student training. With use_real_trainer=true (all leaderboard runs), a fixed modded-nanogpt-style student is trained on the corpus inside the same rollout-owned Docker container or Modal sandbox the agent used, then evaluated on the NanoGPT-speedrun validation stream (FineWeb sample-10BT GPT-2 val tokens, first 10,485,760). A deterministic CPU heuristic stands in during GPU-less development. 
manifest.json ──▶ CorpusBuilder ──▶ curated corpus ──┬──▶ proxy-student training ──▶ Perf ─┐
                                                     │                                      ├──▶ reward
                                                     └──▶ decon screening ──────▶ Leakage ──┘

How this site is built

The leaderboard is a static site generated by docs/build_site.py using the docs/site_builder/ package. collect.py discovers runs in outputs/evals-400m/** (each a config.toml + results.jsonl), filters.py keeps only full 400M real-trainer evals, traces.py extracts the rollout conversation, and debug_runs.py folds in curation snapshots from outputs/debug/. render.py writes data/manifest.json plus one JSON payload per trace and copies the static assets (this page included). The leaderboard and the per-run trace viewer (traces/run.html) are plain HTML/JS reading those JSON files — no server-side code. 

outputs/evals-400m/<run>/{config.toml, results.jsonl} ─┐
outputs/debug/<run>/                                   ├─▶ python docs/build_site.py ─▶ docs/site/
site_builder/assets/ (HTML/CSS/JS)  ───────────────────┘      ├── index.html + data/manifest.json
                                                              ├── codebase.html   (this page)
                                                              └── traces/run.html + data/traces/*.json

Rebuild any time with python docs/build_site.py from environments/pretrain_data_curator/. See the repository README for eval configuration, environment args, and backend prerequisites.
